# Contributing to VID

**Thank you for your interest in improving Verified Intent Development!**

VID is a living methodology that evolves as AI capabilities advance and teams gain real-world experience. Your contributions help the entire community build better practices for AI-augmented development.

---

## How to Contribute

### 1. Share Your Experience

The most valuable contributions come from practitioners:

**Did VID catch a bug?** Share the story
- What principle/practice helped?
- What was the time investment vs. time saved?
- Could become a teaching example

**Did you adapt VID for your context?** Document it
- Industry-specific modifications (fintech, healthcare, etc.)
- Team size variations (solo dev, 100+ eng teams)
- Technology stack adaptations (mobile, embedded, ML, etc.)

**Did you find a gap or limitation?** Report it
- What scenario does VID not address well?
- What principle conflicts with another?
- What practice is unclear or impractical?

### 2. Improve Documentation

**Found something unclear?** Suggest improvements
- Confusing explanations
- Missing examples
- Outdated references

**Created training materials?** Share them
- Workshop slides
- Team onboarding docs
- Case studies

**Developed tools?** Contribute them
- Verification checklists
- Risk assessment calculators
- IDE plugins or scripts

### 3. Expand the Methodology

**New patterns discovered?** Document them
- Verification techniques that work well
- Anti-patterns to avoid
- Risk assessment insights

**Research or data?** Share findings
- Metrics on VID effectiveness
- A/B comparisons
- Academic research

## Contribution Guidelines

### Before Contributing

1. **Read the core methodology** (at minimum: Chapters 1, 4, 5)
2. **Check existing issues/discussions** to avoid duplication
3. **Align with VID principles** (contributions should strengthen, not dilute)

### Types of Contributions

#### Bug Reports / Corrections

**For factual errors, typos, broken links:**

1. Open an issue describing the problem
2. Include:
   - Location (chapter, section, line)
   - Current text
   - Suggested correction
   - Why it's incorrect

#### Content Improvements

**For enhancing existing content:**

1. Open an issue describing the improvement
2. Include:
   - What section needs improvement
   - Why current version is insufficient
   - Proposed changes (draft welcome)

#### New Content

**For new chapters, sections, or appendices:**

1. Open an issue proposing the addition
2. Include:
   - Topic and scope
   - Why it's needed (gap it fills)
   - Target audience
   - Rough outline

3. Wait for community feedback before writing
4. Once approved, submit draft for review

#### Case Studies & Examples

**For real-world VID applications:**

1. Use the template below
2. Anonymize sensitive details
3. Focus on learning value
4. Submit as issue or pull request

**Case Study Template:**
```markdown
## Case Study: [Brief Title]

**Context:** [Industry, team size, tech stack]

**Scenario:** [What happened]

**VID Practice Applied:** [Which principle/practice]

**Outcome:** [What was caught/prevented/learned]

**Time Investment:** [Verification time]

**Time Saved:** [Debugging/incident time avoided]

**Lessons:** [Key takeaways]
```

#### Tools & Automation

**For VID-supporting tools:**

1. Ensure tool aligns with VID principles
2. Document how to use it
3. Include license (prefer MIT/Apache)
4. Link in issue or provide repository

## Contribution Process

### Small Changes (Typos, Links, Formatting)

1. Fork the repository
2. Make changes in your fork
3. Submit pull request with clear description
4. Maintainers will review within 1 week

### Medium Changes (Content Improvements, New Examples)

1. Open an issue first (discuss before writing)
2. Get feedback from maintainers
3. Fork and make changes
4. Submit pull request referencing the issue
5. Incorporate review feedback
6. Merge once approved

### Large Changes (New Chapters, Major Revisions)

1. Open an issue with detailed proposal
2. Community discussion (2+ weeks)
3. Outline approval from maintainers
4. Submit draft in pull request
5. Iterative review process
6. Merge once consensus reached

## Style Guidelines

### Writing Style

**Voice:** Professional but accessible
- Avoid jargon unless necessary
- Define terms when first used
- Use active voice
- Be concise

**Tone:** Practical and evidence-based
- Focus on "why" not just "what"
- Use real examples
- Acknowledge trade-offs
- Avoid dogmatism

**Structure:**
- Start with the problem
- Explain the principle
- Show practical application
- Include examples
- Provide templates/checklists when applicable

### Code Examples

**All code examples should:**
- Be realistic (not toy examples)
- Show both "before" and "after" when relevant
- Include comments explaining key points
- Work if copy-pasted (when appropriate)
- Demonstrate VID principles in action

**Prefer:**
- Common languages (Python, JavaScript, Java, Go)
- Clear variable names
- Minimal dependencies

### Formatting

**Markdown:**
- Use ATX-style headers (`#` not underlines)
- Fence code blocks with language identifiers
- Use tables for structured data
- Keep line length reasonable (<100 chars when possible)

**Diagrams:**
- ASCII diagrams for simple visuals
- Mermaid for complex flowcharts
- Keep accessible (screen reader friendly)

## What We're Looking For

### High Priority

- **Real-world case studies** with measurable outcomes
- **Industry-specific adaptations** (healthcare, fintech, etc.)
- **Metrics and data** on VID effectiveness
- **Edge cases and limitations** we haven't addressed
- **Integration guides** for specific tools/workflows

### Medium Priority

- **Additional examples** for existing practices
- **Improved explanations** of difficult concepts
- **Translation** to other languages
- **Accessibility improvements**
- **Tool integrations** (IDE plugins, CI/CD, etc.)

### Low Priority (But Still Welcome!)

- **Minor wording improvements**
- **Additional references** to related work
- **Formatting enhancements**

---

## What We're NOT Looking For

### Out of Scope

- **Prompt engineering guides** (VID is about verification, not generation)
- **AI model comparisons** (VID is model-agnostic)
- **Productivity hacking** (VID is about correctness, not speed)
- **Generic coding best practices** (unless specific to AI-generated code)
- **Religious debates** about AI (focus on practical methodology)

### Against VID Principles

- **Shortcuts that reduce verification**
- **"Trust the AI" approaches**
- **Eliminating understanding requirements**
- **Risk calibration by convenience rather than actual risk**

## Review Process

### Timeline

- **Small changes:** 1 week review
- **Medium changes:** 2-3 weeks review + iteration
- **Large changes:** 4-8 weeks discussion + review

### Review Criteria

Contributions are evaluated on:

1. **Alignment with VID principles**
   - Does it strengthen verification practices?
   - Does it maintain appropriate skepticism?

2. **Practical value**
   - Is it actionable?
   - Does it solve a real problem?
   - Can teams actually use this?

3. **Quality**
   - Is it well-written?
   - Are examples realistic?
   - Is evidence provided?

4. **Scope**
   - Does it fit within VID's focus?
   - Is it the right level of detail?

### Feedback

- All feedback aims to be constructive
- Rejections explain why
- Revisions may be requested
- Community input is valued

## Community Standards

### Code of Conduct

**Be respectful:**
- Focus on ideas, not people
- Assume good intentions
- Disagree professionally

**Be constructive:**
- Provide rationale for disagreement
- Suggest alternatives
- Help improve proposals

**Be collaborative:**
- Build on others' ideas
- Give credit where due
- Share knowledge generously

**Stay focused:**
- Keep discussions on-topic
- VID methodology only (not general AI debates)
- Practical over philosophical

### Attribution

- Original authors are credited
- Contributors are acknowledged
- Ideas are attributed to sources
